# User Story UC015.005: Classify Transaction (Category / Subcategory)

| Attribute | Value |
|-----------|-------|
| **Story ID** | UC015.005 |
| **Epic** | Financial Management |
| **Related UC** | UC015 |
| **Priority** | MUST HAVE |
| **Story Points** | 3 |

**As an** ADMIN **I want to** assign a category and subcategory to a transaction **so that** it can be used in statistical analysis and reporting.

## Acceptance Criteria

**AC1:** Classification section present in both the create form (UC015.002) and the edit form (UC015.003).

**AC2:** Category dropdown shows all categories ordered alphabetically. No direction filtering on categories (categories are shared).

**AC3:** Subcategory dropdown shows only subcategories belonging to the selected category AND compatible with the transaction direction (subcategory.direction IN [transaction direction, BOTH]). Ordered alphabetically.

**AC4:** Subcategory cannot be selected before a category is chosen. If category is cleared, subcategory is also cleared.

**AC5:** A transaction can have at most one subcategory (and therefore one category).

**AC6:** Classification is optional — a transaction may be saved without a subcategory (e.g. during import before review).

**AC7:** On save, if both `subcategory` and `counterparty_account` are set:
- System looks up existing `tag_learning_rule` for `(COUNTERPARTY_ACCOUNT, counterparty_account, subcategory_id)`.
- If found: `confidence += 1`, `last_matched_at = now`.
- If not found: new rule created with `confidence = 1`.

**AC8:** On save, if both `subcategory` is set and `accounting_month` differs from the month of `transaction_date`:
- Offset = number of months between `transaction_date` month and `accounting_month`.
- System looks up existing `accounting_month_rule` for `(subcategory_id, counterparty_account)`. If `counterparty_account` is empty, matches on `(subcategory_id, NULL)`.
- If found: `confidence += 1`, `last_matched_at = now`.
- If not found: new rule created with `confidence = 1`.

**AC9:** Applying a subcategory whose direction is incompatible with the transaction direction → backend returns HTTP 400 with message "Subcategory '[name]' is not compatible with direction [direction]." (BR-UC015-06).

**Endpoint:** Classification is part of `POST /api/v1/transactions` and `PUT /api/v1/transactions/{id}`.
