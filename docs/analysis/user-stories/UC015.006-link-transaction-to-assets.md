# User Story UC015.006: Link Transaction to Asset(s)

| Attribute | Value |
|-----------|-------|
| **Story ID** | UC015.006 |
| **Epic** | Financial Management |
| **Related UC** | UC015 |
| **Priority** | SHOULD HAVE |
| **Story Points** | 3 |

**As an** ADMIN **I want to** link an expense transaction to one or more physical assets **so that** I can trace maintenance and service costs per asset over time.

---

## Context

A transaction has two independent linking mechanisms that coexist:

**1. Contextual link** (existing, unchanged) — used for all transaction types:
- Building (optional)
- Housing unit (optional, cascades from building)
- Lease (optional, INCOME only)

**2. Asset links** (this story) — optional, EXPENSE only:
- One or more physical devices (boiler, fire extinguisher, meter)
- Housing unit and building are **automatically derived** from the selected device and displayed read-only
- A partial amount can be assigned per device to split a single invoice across multiple assets

These two mechanisms are independent. A rent transaction uses only mechanism 1. A boiler maintenance invoice may use both (building in mechanism 1, individual boilers in mechanism 2) or only mechanism 2.

---

## Acceptance Criteria

**AC1:** The Asset Links section is visible **only when `direction = EXPENSE`**. It is entirely hidden for INCOME transactions. Using the section is optional — an EXPENSE transaction can be saved without any asset link.

**AC2:** "Add Asset Link" button adds a new row with: asset type selector (BOILER / FIRE_EXTINGUISHER / METER), asset picker, amount field (optional), notes field (optional).

**AC3:** The asset picker is a **text search** (by device identifier, brand, unit number) and is filtered by the building already selected on the transaction if one is set. If no building is set on the transaction, the picker searches across all buildings.

**AC4:** Once a device is selected, the **housing unit and building are automatically resolved** from the device's existing relationships and displayed read-only on the row. The admin does not enter these manually:
- BOILER → `boiler.housing_unit_id` → unit + building
- FIRE_EXTINGUISHER → `fire_extinguisher.unit_id` (nullable) + `fire_extinguisher.building_id` → unit (if any) + building
- METER → `meter.owner_id` + `meter.owner_type` → unit or building

**AC5:** Asset picker content depends on asset type and context:
- **BOILER**: all active boilers of the filtered context (building if set, otherwise all)
- **FIRE_EXTINGUISHER**: all extinguishers of the filtered context
- **METER**: all active meters of the filtered context
- Each result shows enough context to identify the device: brand/model or identifier + unit number + building name

**AC6:** The `amount` field on each row is optional:
- If **one row only** and no amount entered: the full transaction amount is implicitly attributed to that asset
- If **multiple rows**: entering partial amounts is encouraged; a running total is shown below the rows
- The sum of partial amounts must not **exceed** the transaction total (validation error if it does)
- The sum may be **less than** the transaction total (partial attribution is allowed)

**AC7:** A ventilation summary is shown below the asset link rows:
```
Ventilated: 1 200.00 € / Transaction total: 1 200.00 € ✓
```
Color coded: green if sum ≤ total, red if sum > total.

**AC8:** When the asset type is selected, the **subcategory is automatically pre-filled** if a mapping exists in platform config (`asset.type.subcategory.mapping.*`). The admin can override this suggestion freely. Examples:
- `BOILER` → Maintenance / Chaudière
- `FIRE_EXTINGUISHER` → Maintenance / Extincteurs

**AC9:** Multiple asset links are allowed on a single transaction. The same asset can appear **only once** per transaction → duplicate asset → error "This asset is already linked to this transaction."

**AC10:** Asset link for type BOILER where the boiler does not belong to any unit of the transaction's building → backend returns HTTP 400: "The selected boiler does not belong to the transaction's building." (BR-UC015-09). This rule applies only when a building is explicitly set on the transaction.

**AC11:** Each row has a "×" remove button (no confirmation required).

**AC12:** In the boiler details section (housing unit details page): a "Related expenses (N)" badge is shown. Clicking it navigates to the transaction list filtered to `assetType=BOILER&assetId={id}`.

**AC13:** Same badge in the fire extinguisher section on building details page.

**AC14:** Asset links are saved as part of the transaction create/update payload. No separate endpoint.

## Endpoint

Asset links embedded in `POST /api/v1/transactions` and `PUT /api/v1/transactions/{id}` payloads.

Each asset link in the request payload:
```json
{
  "assetType": "BOILER",
  "assetId": 12,
  "amount": 400.00,
  "notes": "révision annuelle"
}
```

`housingUnitId` and `buildingId` are **not** sent by the frontend — they are resolved server-side from the device relationship.
