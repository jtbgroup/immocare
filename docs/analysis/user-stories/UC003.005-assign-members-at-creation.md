# UC003.005 — Assign Members at Estate Creation

## Overview

| Attribute | Value |
|---|---|
| **ID** | UC003.005 |
| **UC** | [UC003 — Manage Estates](../use-cases/UC-003-manage-estates.md) |
| **Actor** | PLATFORM_ADMIN |
| **Priority** | MUST HAVE |

## User Story

**As a** PLATFORM_ADMIN **I want to** assign multiple members with their roles when creating an estate **so that** the estate is immediately usable without a separate membership step.

## Acceptance Criteria

- AC1: The `members` field of `CreateEstateRequest` accepts a list of `{ userId, role }` entries.
- AC2: Each `userId` must reference an existing `app_user` → HTTP 404 if not found.
- AC3: If the list is non-empty, at least one entry must have role `MANAGER` → HTTP 422 if violated.
- AC4: Members can also be added separately after creation via `POST /api/v1/estates/{id}/members` (UC003.007).
- AC5: The PLATFORM_ADMIN is not automatically added as a member — they access all estates via their global flag.
- AC6: All members are persisted atomically with the estate in a single transaction.
- AC7: In the UI, the creation form provides a searchable user picker and role selector. The user builds the member list interactively before submitting.
- AC8: A warning is shown in the UI if members are added but none is assigned the MANAGER role.

## Endpoint

Handled as part of `POST /api/v1/admin/estates` (see UC003.001).

## Business Rules

- BR-UC003-02: At least one MANAGER required when a non-empty members list is provided
- BR-UC003-05: PLATFORM_ADMIN does not need a member entry to access estates
- BR-UC003-15: Duplicate userIds in the list are deduplicated silently
